# Design: `dictionary` command — type-safe translations

Date: 2026-06-26
Status: Approved (pending spec review)

## Goal

Add a standalone CLI command that fetches the Sitecore dictionary from Experience
Edge and generates type-safe translation artifacts, so call sites using `t()`
get autocomplete and compile-time errors on unknown keys.

## Decisions (settled during brainstorming)

- **Data source:** fetch from Experience Edge via the Content SDK dictionary
  GraphQL query (same site/language/auth as the existing layout fetch). Not parsed
  from `_app.tsx` (that is runtime data) and not read from a local JSON file.
- **Command shape:** a new standalone command `dictionary` (the dictionary is
  site-global, not per-route — so it is not folded into `page`).
- **t() integration:** target `next-localization`'s `useI18n().t`. Generate a typed
  `useTypedT()` wrapper hook that constrains keys to the generated union.
- **Key representation:** const map + derived type — runtime key references AND the
  type, avoiding magic strings at call sites.
- **Rebuild model:** one idempotent command. Re-running resyncs the keys; the
  wrapper is never clobbered. No separate init/sync split.
- **Output dir:** default `src/lib/i18n` (configurable).
- **Source language:** keys generated from a single language (the default or
  `--lang`), since dictionary keys are language-invariant.

## Architecture

Mirrors the existing layout flow (Edge query → fetch → normalize → codegen),
split across the `core` and `cli` packages exactly as the current commands are.

### `@sitecore-scaffold/core` additions

- `edge/query.ts` — add `DICTIONARY_QUERY`: the paginated
  `site.siteInfo(site).dictionary(language, first, after)` query returning
  `results { key value }` and `pageInfo { endCursor hasNext }`.
- `edge/client.ts` — add `getDictionary(language)` that **follows pagination**
  (loops on `pageInfo.hasNext`/`endCursor`) and returns
  `Array<{ key: string; value: string }>`. Same error handling as `getLayout`
  (HTTP errors, GraphQL `errors[]`).
- `dictionary/build.ts` — normalize raw entries: dedupe keys and sort them
  alphabetically (stable diffs). Returns `{ keys: string[]; warnings: string[] }`
  (warns when the dictionary is empty).
- `codegen/dictionary-file.ts` — produce the `dictionary-keys.ts` `GeneratedFile`.
- `codegen/typed-t-file.ts` — produce the `use-typed-t.ts` `GeneratedFile`.
- Export the new functions/queries from `core/src/index.ts`.

### `@sitecore-scaffold/cli` additions

- `args.ts` — accept `dictionary` as a valid command (reuses `--lang`,
  `--dry-run`, `--force`; no positional required). Update `USAGE`.
- `commands/dictionary.ts` — `runDictionary` mirroring `runPage`: load config,
  fetch dictionary, build, generate, write (honoring dry-run/force/skip rules).
- `index.ts` — dispatch `dictionary` and print a summary (files generated /
  skipped / warnings), consistent with the `page` summary.

## Generated artifacts

### `<i18nPath>/dictionary-keys.ts` — always overwritten

```ts
// AUTO-GENERATED by sitecore-scaffold. Do not edit.
export const dictionaryKeys = {
  'Home.Title': 'Home.Title',
  'Nav.Login': 'Nav.Login',
} as const;

export type DictionaryKey = keyof typeof dictionaryKeys;
```

Keys are emitted as **quoted string-literal** object keys, so keys containing dots
or spaces need no identifier sanitization and the type union stays exact.

### `<i18nPath>/use-typed-t.ts` — scaffolded once (skipped if exists, unless `--force`)

```ts
import { useI18n } from 'next-localization';
import type { DictionaryKey } from './dictionary-keys';

export function useTypedT() {
  const { t } = useI18n();
  return (key: DictionaryKey, params?: Record<string, unknown>, lang?: string) =>
    t(key, params, lang);
}
```

The `next-localization` import path comes from the `i18nPackage` config option.

Usage at call sites:

```ts
const t = useTypedT();
t(dictionaryKeys['Nav.Login']); // autocompleted; unknown keys are type errors
```

## Regeneration behavior

- `dictionary-keys.ts` is a **pure generated artifact → always overwritten** so it
  tracks the live dictionary as entries come and go.
- `use-typed-t.ts` is a **scaffold the user may customize → written once, skipped if
  present** unless `--force` (same rule as component files).
- Re-running `dictionary` is therefore the rebuild path: it resyncs the keys and
  leaves a customized wrapper intact.

## Config additions

In `ScaffoldConfig` (`core/src/types.ts`), `load.ts` defaults, and the example file:

- `i18nPath: string` — default `src/lib/i18n`.
- `i18nPackage: string` — default `next-localization`.

## Error / edge cases

- Empty dictionary → emit a warning; still write a valid (empty) `dictionaryKeys`
  map and type rather than failing.
- Duplicate keys across pagination pages → deduped in `build`.
- Edge HTTP / GraphQL errors → surfaced the same way as `getLayout`.
- Missing route is not relevant (no route argument).

## Testing (vitest, matching existing patterns)

- `dictionary/build` — dedupe, alphabetical sort, empty-dictionary warning.
- codegen — snapshot both generated files; verify generated code typechecks via the
  existing `test/helpers/typecheck.ts` helper.
- `edge/client.getDictionary` — mock fetch; assert pagination is followed across
  multiple pages and entries are concatenated.
- `commands/dictionary` — dry-run prints both files; write creates files; wrapper is
  skipped when it already exists; `--force` overwrites the wrapper; keys file is
  always overwritten.

## Out of scope

- Multi-language value snapshots / default fallback dictionary in source.
- react-i18next module augmentation (chosen integration is next-localization).
- Folding dictionary generation into the `page` command.
- A separate init/sync command split.
